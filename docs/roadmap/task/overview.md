# Lime 任务层 / 模型层设计总览

> 状态：提案
> 更新时间：2026-04-23
> 作用：先把为什么要拆“任务层 / 模型层”、如何兼容单模型与多模型、为什么要处理 `service_models` 和设置平衡讲清楚。
> 依赖文档：
> - `./README.md`
> - `./task-taxonomy.md`
> - `./model-routing.md`

## 1. 背景

Lime 现在已经有多条与模型选择相关的输入面：

1. 会话中的聊天模型选择
2. `workspace_preferences.service_models`
3. 媒体生成默认模型与 companion 默认模型
4. `service_scene_launch` / 工作台 `request_metadata`
5. OEM `providerPreference/defaultModel` 与 offer
6. 运行时会话恢复得到的 `provider_name / model_name`

问题不在于“缺少模型选择入口”，而在于：

- 每条入口都只解决自己那一小块
- 没有统一表达“这次是什么任务”
- 没有统一表达“当前真实可选空间是什么”
- 没有统一处理“只有一个模型怎么办”
- 没有把成本/限额沉到底层 runtime

## 2. 对 Claude Code 的借鉴，不是简单照搬

从对照参考运行时可以借鉴的，是下面这些原则：

| 借鉴点 | Claude Code 的做法 | Lime 应该借鉴什么 |
| --- | --- | --- |
| 主任务模型与辅助任务模型分层 | 主循环模型独立，小任务会走小模型 | 把“任务画像”和“主对话选择模型”拆开 |
| 成本/限额是底层事件 | `rate_limit_event`、`total_cost_usd`、`modelUsage` | 成本/限额必须写进 runtime / telemetry / thread read |
| 小任务可自动用小模型 | 标题、摘要、部分 hooks、部分 search 包装 | 内部服务任务可先自动调度，主对话再渐进开放 |
| 子代理和任务配置允许带模型语义 | 任务可声明 alias，但仍属于任务层 | Lime 要把 `service_models` 和任务画像打通 |

Lime 不能直接照搬的点：

1. Lime 有 OEM 控制面。
2. Lime 既可能只有一个模型，也可能同时有多个 provider/model。
3. Lime 已经存在多种设置入口，不能假装这些不存在。

## 3. 为什么即使只有一个模型，也要拆任务层和模型层

这是本专题最重要的固定结论。

即使当前只有一个模型，系统仍然需要知道：

1. 这次是主对话、翻译、压缩、标题生成，还是浏览器研究任务。
2. 当前唯一模型是否支持所需能力。
3. 当前唯一模型是否成本过高、配额不足、并发受限。
4. 当前任务是否应该走 `service_models` 设置的服务模型。
5. 当前任务是否必须服从 OEM 指定模型。

所以：

- **任务层**回答“要做什么”
- **模型层**回答“当前能不能做、怎么做、是否要降级”

当只有一个模型时，模型层会退化成：

- 候选集解析
- 能力校验
- 成本/限额判断
- 降级解释

它不会消失。

## 4. 固定三类运行时现实

后续所有设计都必须同时支持这三类现实：

### 4.1 单候选现实

当前只有：

- 一个 key
- 一个 provider
- 一个模型
- 或一个 OEM session

这时系统只能做：

- 识别任务类型
- 校验能力
- 标注成本档
- 判断是否受限
- 可解释降级

### 4.2 多候选现实

当前有多个 provider/model 候选。

这时系统才能做：

- 在设置与约束允许范围内自动优选
- 在成本、能力、限额之间平衡
- 做 fallback chain

### 4.3 任务级设置现实

Lime 已经存在 `service_models`，说明不同服务任务本来就需要不同模型偏好。

这意味着：

- 不能再把所有任务都压成“会话当前模型”
- 也不能让 `service_models` 永远绕过统一 runtime 路由

## 5. 自动与设置如何平衡

本专题固定采用“三层优先级”：

### 5.1 硬锁定

包括：

- 用户本轮显式 override
- 用户把当前会话模型明确锁定
- OEM `managed` 硬约束

硬锁定优先，自动策略不能越过。

### 5.2 任务偏好

包括：

- `service_models`
- `service_scene_launch` 的首选 provider/model
- 某类服务任务的默认模型

任务偏好是强提示，但不是无条件硬锁。若配置失效、限额不足或能力不匹配，系统应可解释回退。

### 5.3 自动优化

只有在没有硬锁定、且任务偏好允许回退、且候选集大于 1 时，自动优化才真正生效。

## 6. 本专题引入的三个核心对象

### 6.1 `TaskProfile`

描述这次任务是什么、需要什么能力、延迟目标和预算档。

### 6.2 `CandidateModelSet`

描述当前 runtime 在真实约束下到底有哪些可用候选，而不是“理论上模型目录里有哪些模型”。

### 6.3 `RoutingDecision`

描述最后为什么选这个模型，或者为什么没有选、为什么只能降级。

## 7. 这套设计对现有 Lime 设置面的含义

### 7.1 聊天模型选择器

继续保留，但角色变成：

- 显式锁定
- 会话偏好
- 自动模式的候选起点

### 7.2 `service_models`

继续保留，但角色升级成：

- 任务级偏好输入源
- 内部服务任务的首选 provider/model
- 不是绕过 runtime 的旁路

### 7.3 OEM `providerPreference/defaultModel`

继续保留，但角色是：

- OEM 控制面的约束或建议
- 不是前端直接认定的最终真实模型

## 8. 本专题的非目标

这套路线图当前不试图解决：

1. 所有 provider 的真实在线价格同步
2. 所有工具调用都自动切小模型
3. 一次性重写现有模型选择 UI
4. 重新定义 Query Loop 主链

## 9. 这一步如何服务主线

这一步最直接的主线收益是：

**让 Lime 能在“单模型能解释、多模型能优化、任务设置能复用、自动策略有边界”这四件事上同时成立。**
