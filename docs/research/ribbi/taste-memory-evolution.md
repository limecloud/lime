# Ribbi 的 Taste / Memory / Evolution 机制

> 状态：current research reference  
> 更新时间：2026-04-18  
> 目标：拆清 Ribbi 为什么强调 taste layer、为什么要做文本化中间态，以及这套机制对 Lime 的直接启发。

## 1. 先区分三个对象

Ribbi 把容易混在一起的三类东西拆开了：

1. `memory`
   - 用户历史、项目历史、交互轨迹、反馈记录
2. `taste`
   - 用户偏好的风格、质感、节奏、审美倾向
3. `feedback`
   - 发布结果、失败信号、显式正负反馈、行为信号

它们会互相影响，但不是同一个对象。

## 2. 为什么 taste 不直接喂图片

Ribbi 的核心判断是：

1. 图片本身信息量大，但 token 成本也大
2. 当用户长期积累很多参考素材时，原始多模态上下文容易爆炸
3. 真正有价值的不是“把原图全塞进去”，而是把品味抽成可压缩的中间态

因此它选择：

1. 先用视觉模型理解参考素材
2. 再提炼成更稳定的文本化风格描述
3. 再把这份描述压进 taste layer

固定判断：

**taste layer 的关键不是保留原始素材，而是保留可复用的风格状态。**

## 3. taste 和 memory 的分工

可以用下面这张表理解：

| 对象 | 回答什么问题 | 典型来源 |
|------|------|------|
| `memory` | 你做过什么、说过什么、项目走到哪 | 历史对话、项目结果、行为轨迹 |
| `taste` | 你觉得什么好看、什么顺眼、什么像你 | 收藏图片、视频、风格参考 |
| `feedback` | 这次结果好不好、该往哪继续迭代 | 运行结果、发布数据、显式判断 |

## 4. 为什么需要 async evolution agents

这些数据如果都在主聊天链上同步完成，会有两个问题：

1. 主执行链会变重
2. 用户会感到很多“不属于当前任务”的处理延迟

因此 Ribbi 选择让异步 agent 在后台持续做：

1. taste 提炼与压缩
2. memory 总结与更新
3. feedback 回写
4. 用户重复行为抽象成可复用 skill

这样主 Agent 仍然只做一件事：

**接住当前创作任务。**

## 5. 新用户冷启动靠什么

Ribbi 在访谈里表达得很清楚：新用户不应该因为还没积累个人 taste 就直接掉到低质量底线。

它的做法是：

1. 平台有一层共享审美基线
2. 新用户先用这层基线兜底
3. 用户一旦开始积累自己的参考素材，系统就逐渐转向个体化 taste

这个机制的意义不是把所有人做得越来越像，而是：

**先保证不难看，再逐渐变得更像你。**

## 6. 对 Lime 的术语映射

Lime 后续对照 Ribbi 时，统一使用下面这组翻译，不再引入 Ribbi 的收藏池命名：

| Ribbi 常见表达 | Lime 对照口径 |
|------|------|
| Ribbi 的收藏层 | `灵感库` 的前台收藏与参考入口 |
| taste pool | `Taste Layer` 内部风格摘要 |
| saved inspirations | `灵感对象 / 参考条目` |
| taste agent | `异步风格提炼 agent` |

固定规则：

1. Lime 不用 Ribbi 的收藏池命名
2. 前台统一讲 `灵感库`
3. 后台统一讲 `Taste Layer`

## 7. 对 Lime 的直接启发

Lime 后续最应该对齐的不是“做一个同名收藏池”，而是：

1. 让灵感库成为 taste / reference / memory 的统一前台投影
2. 让 taste 成为正式对象，而不是聊天里的描述文字
3. 让反馈与复盘结果继续反哺下一轮 planning
4. 让 taste / memory 的更新尽量异步，不阻塞 `生成` 主执行面
