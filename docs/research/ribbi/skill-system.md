# Ribbi 的 Skill 系统

> 状态：current research reference  
> 更新时间：2026-04-18  
> 目标：说明 Ribbi 的 skill 为什么不是“功能列表”，以及 Lime 后续应该怎样借鉴它。

## 1. 固定判断

Ribbi 的 skill 不是：

1. 一个个独立小工具
2. 死板 workflow
3. 单纯 prompt 模板
4. 让用户自己理解底层模型的入口

Ribbi 的 skill 更像：

**一份面向场景的说明书，告诉主 Agent 这类事情通常怎么做。**

## 2. skill 的五个组成部分

一个完整的 Ribbi 式 skill，至少包含五部分：

1. 场景名
   - 用用户结果命名，而不是底层能力命名
2. 输入槽位
   - 告诉系统这类任务需要哪些最小输入
3. 工具建议
   - 哪些工具通常会被用到
4. 结果约定
   - 产物应该长成什么样
5. 审美或风格约定
   - 某些 skill 自带默认审美或表达调性

因此，skill 的本质不是按钮，而是“有边界的任务语义包”。

## 3. skill 和 workflow 的区别

两者最核心的区别只有一句：

**workflow 是强约束，skill 是弱建议。**

workflow 的问题：

1. 先把顺序写死
2. 一旦用户需求超出设计路径，就容易卡住
3. 用户只能在预设线性路径里思考

skill 的优点：

1. 提供默认做法，但不限制模型重新组合工具
2. 能和其他 skill 连通
3. 更贴近自然语言任务

## 4. skill 的命名方式

Ribbi 值得学的不是“skill”这个词，而是它的命名方法。

好的命名应该满足三条：

1. 说清用户要完成什么
2. 说清输入大概是什么
3. 不暴露工具或模型细节

好例子：

1. 每日趋势摘要
2. 复刻一条短视频
3. 把内容变成其他语言
4. 追踪账号表现

坏例子：

1. 调用视频模型
2. 运行内容工作流
3. 触发社媒 agent
4. 启动多模态编排

## 5. skill 前台化与后台化的分界

应该前台化成 skill 的，通常是：

1. 用户能直接理解的任务
2. 结果导向清晰的创作动作
3. 高频、可重复、可沉淀的内容流程

不应该前台化的，通常是：

1. 底层工具
2. provider 选择
3. 中间格式转换
4. 单步技术操作

一句话：

**前台显示任务，后台吞掉工具。**

## 6. 对 Lime 的借鉴原则

Lime 后续做 skill-first 设计时，应该直接继承以下判断：

1. skill 是场景说明书，不是功能按钮集合
2. skill 卡片应该少而强，不追求一次性铺满目录
3. skill 的输入应该是少量槽位，而不是期待用户先写一大段 prompt
4. skill 应该指向 `生成` 主执行面，而不是再开第二套执行壳
